# Skill: manual-writing

## Cuándo usar esta skill

Usa esta skill cuando la tarea sea:
- redactar un manual de usuario
- transformar análisis funcional en contenido visible al cliente
- escribir tutoriales de uso
- organizar secciones de guía
- limpiar lenguaje interno o demasiado técnico
- convertir documentación híbrida en documentación final

## Objetivo

Escribir contenido final para clientes y usuarios, con tono:
- claro
- profesional
- institucional
- práctico
- final
- listo para publicarse

## Reglas obligatorias

- Escribir siempre en español claro.
- Priorizar cliente final.
- No exponer:
  - dashboards base usados para construir la guía
  - nombres de archivos fuente
  - comparativas internas
  - validaciones editoriales internas
  - cocina funcional de PARKS
- No usar tono de “sitio en construcción”.
- No usar frases tipo:
  - guía publicada
  - módulo destacado
  - base documental
  - complementos técnicos
  - futuras ampliaciones como justificación de contenido faltante
- Si se menciona una duda funcional, hacerlo solo si es realmente útil para el usuario final.
- No inventar funciones que el usuario no tenga disponibles.

## Cómo escribir

### Debe sonar así
- oficial
- claro
- útil
- orientado a la acción
- centrado en lo que el usuario ve y hace

### No debe sonar así
- técnico interno
- auditoría
- relevamiento
- trabajo en progreso
- proceso editorial
- desarrollo incompleto

## Estructura preferida para manuales

- Introducción general
- Objetivo
- Cómo navegar
- Cómo usar filtros
- Cómo interpretar elementos
- Cómo interactuar
- Buenas prácticas de uso
- Secuencia recomendada de lectura
- Cierre o resumen

## Si el manual es de Power BI

Explicar:
- navegación del dashboard
- filtros y fechas
- KPIs y tarjetas
- gráficos
- tablas
- mapas
- interacciones
- pestañas principales
- buenas prácticas de uso

No explicar:
- estructura interna de PBIX
- arquitectura de dataset
- lógica de modelado
- diferencias internas entre dashboards salvo que sean útiles para el cliente

## Reformulaciones preferidas

### Usar en vez de lenguaje interno
- “Buenas prácticas de uso”
- “Interacciones disponibles”
- “Secuencia recomendada de lectura”
- “Cómo interpretar las pestañas del dashboard”

### Evitar
- “Errores comunes de interpretación” si suena acusatorio
- “Elementos que pueden requerir validación” si eso es trabajo interno
- “Espacio para futuras ampliaciones”
- “Módulos o pestañas que pueden aparecer” si genera incertidumbre innecesaria

## Formato sugerido

Usar encabezados claros y secciones cortas.
Si el texto es largo:
- partirlo en bloques
- agregar listas breves
- incluir bloques de consejo, advertencia o recomendación solo si aportan valor real

## Regla final

Si una frase ayuda al equipo interno pero no al cliente, eliminarla o reescribirla.
